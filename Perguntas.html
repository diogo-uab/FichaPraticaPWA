Ficha prática para treino de manuseamento de conteúdos em repositórios Github

  1. O que são operações de “Commit”, “Push” e “Pop” efetuadas num repositório?

    • A operação Commit consiste em guardar alterações num repositório. Operação essa que pode conter, por exemplo, alterações em ficheiros existentes, adicionar novos ficheiros ou remover ficheiros de um repositório.

    • A operação Push permite enviar, ou puxar, atualizações presentes num repositório local para um repositório remoto (por exemplo, um repositório no GitHub).

      Os commits são um bom exemplo das alterações que podem ser puxadas para um repositório remoto, fazendo com que este fique ao mesmo nível do repositório local.

    • A operação Pop, ou stash pop, consiste em remover alterações locais que foram previamente guardadas na lista de stash e em adicioná-las ao repositório local no estado em este que se encontra.

  2. Quais são as vantagens de usar Branches num repositório git?

    O uso de Branches num repositório git proporciona muitas vantages, especialmente no que toca à colaboração com vários elementos de uma equipa no mesmo repositório. Algumas dessas vantagens incluem:

    • Padronização de um "Gitflow", com processos e nomenclaturas claras seguidas por toda a equipa, o que facilita processos como o desenvolvimento de novas funcionalidades, lançamentos de novas versões e resolução de erros.

    • Eficiência na colaboração, permitindo que vários membros trabalharem em simultanêo no mesmo repositório com a mínima fricção.

    • Facilita tanto a visão geral de alterações com operações como o Diff, como a revisão das mesmas por outros membros da equipa, recorrendo por exemplo à criação de um "Pull Request" (este nome pode variar entre plataformas, mas é provavelmente o mais conhecido devido ao seu uso no Github).

    • Flexibilidade para trocar contexto, por exemplo passar do desenvolvimento de uma nova funcionalidade para a resolução de erro crítico em produção com facilidade.

  3. Nos Merges de conteúdos resultantes de um “Pop” podem resultar em conflitos. Em que consistem e como podem ser resolvidos esses conflitos?

    Os conflitos podem acontecer quando a mesma região de um ficheiro foi alterada em mais do que um branch, ou presente num stash como neste caso, e o git não consegue decidir automaticamente que versão da alteração deve manter.

    Estes conflitos podem ser manualmente resolvidos em todos os blocos alterados e em conflito, tanto escolhendo quais alterações devem ser mantidas entre as que estão presentes no branch atual (HEAD) ou as do branch relativo ao merge ou stash, como uma combinação de ambas as alterações.

  4. “Clone” e “Fork” são formas diferentes de replicar um repositório Git. Quais as diferenças entre eles?

    Embora ambas as operações Clone e Fork repliquem um repositório git, a forma como essa réplica é feita e utilizada, assim como a sua relação com o repositório original são bastante diferentes.

    • A operação Clone replica um repositório git remoto para uma máquina local.

      Permitindo assim trabalhar localmente no repositório e interagir diretamente com o repositório original, por exemplo, através da operação Push.

    • Já a operação Fork cria uma réplica independente do repositório, geralmente numa platforma como o Github. Essensialmente, um novo repositório com base mas desconectado do repositório original.

      É uma operação que permite, por exemplo, contribuir em repositórios sem afetar o original, podendo mais tarde criar "Pull Requests" para o repositório de base do fork.

      Permite também utilizar o repositório original como um ponto de partida para a criação de um projeto novo, sem afetar o repositório original.
